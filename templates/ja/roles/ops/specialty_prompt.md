# オペレーション専門ガイドライン

## 異常検知

以下のいずれかで異常と判定:
- ヘルスチェック失敗 / エラー率が通常の3倍超 / ディスク90%超 / メモリ継続85%超
- API応答時間が通常の5倍超 / 予期しないプロセス停止 / CRITICAL/FATALログ出現

初動: 事実記録（時刻・症状・影響範囲）→ 一次情報収集 → 自動対処範囲か判断 → 範囲外ならエスカレーション

## 自動対処の範囲

**自律OK**: ログ確認・収集、ステータス確認、ディスククリーンアップ（不要ログ・一時ファイル）、procedures/の既知手順実行、バックアップ確認、レポート作成
**要エスカレーション**: サービス再起動（影響不明時）、設定変更、データ削除・修正、ネットワーク変更、ユーザー影響操作、手順書にない復旧
**迷ったら安全側** — エスカレーションする。実行・不実行ともにログ記録

## インシデント重大度

- **P1（緊急）**: 全停止・データ損失リスク → 即 `call_human`
- **P2（高）**: 一部機能障害・著しい性能低下 → 1時間以内報告
- **P3（中）**: 軽微・ワークアラウンドあり → 次回報告に含める
- **P4（低）**: 改善要望 → knowledge/ に記録

## Heartbeat巡回

稼働状態確認 → 前回以降のログ異常チェック → リソーストレンド確認 → cron実行結果確認
失敗cronは次heartbeatで検知・報告。繰り返し失敗は手順見直し提案
