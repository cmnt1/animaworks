## machine への依頼内容
以下を**証拠付きで**調べ、短いレポートにまとめること。

### A. 誤警報かどうか
- `0 10 * * 1,4` の schedule と 2026-05-15 07:48 JST 時点を照合し、直近3時間に実行が期待されるかを明示
- `core/schedule_parser.py` の `parse_schedule()` も確認し、この cron 式が正常に解釈される前提に問題がないか確認
- 結論を `誤警報` / `実不具合` / `追加確認要` のいずれかで示す

### B. scheduler 登録状態
- `animaworks.log` から mikoto の scheduler 起動・cron 登録・schedule reload の痕跡を探す
- 見つからない場合は「ログ不足」なのか「実際に未登録の疑い」なのかを、関連コードも見て判断する
- 可能なら `cron_logs` や `activity_log` から「過去に mikoto cron が走っていた証拠」を拾い、登録実績の有無を示す

### C. 警告が2回出た理由
- heartbeat が同一 warning file を2回取り込む経路、または notification file が複数生成される経路がないか、関連コードを確認
- `background_notifications` の消費ロジック、scheduler 側の warning 書き込みタイミング、重複通知の可能性を調べる
- もっともありそうな重複原因を「コード位置 + 推論根拠」で示す

### D. 最小修正案（実装しない）
必要なら、最小修正案を 1〜3 個に絞って提案すること。例:
- ヘルスチェックを「次回予定時刻ベース」にする
- 長周期 cron を health 対象から除外する
- notification 消費側で重複抑止する
